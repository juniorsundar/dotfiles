# Pi-native Deep Research Extension

## Problem Statement

Pi sometimes needs external, decision-relevant research while helping with architecture, technology feasibility, library comparisons, provider behavior, benchmarks, pricing, recent API changes, and alternatives. The current chat context is a poor place to run deep research because intermediate search results, source excerpts, and synthesis notes can overwhelm the context window, disappear after interruption, or become hard for the user to audit.

The user wants deep research to be bounded to a single Research Question, varied in Evidence Mix, inspectable, resumable, and useful both to humans and to the coding agent. The current Tongyi DeepResearch model is available through Ollama, but its setup must not be assumed to work: live testing showed that prompt formatting, stop behavior, inline thinking, and structured output parsing require validation and guardrails.

## Solution

Build a pi-native Research Orchestrator extension. The orchestrator creates editable Research Proposals, obtains explicit user approval, enforces Research Budgets, uses Tongyi DeepResearch as the single Research Brain, stores Research Run artifacts in the Workspace Research Store, emits compact Progress Digests, and produces Research Briefs with traceable citations.

The system should optimize for bounded, source-grounded decision support rather than a long-lived knowledge base. Human-initiated runs should produce a self-contained Human Research View, while agent-triggered runs should include implications for Pi or the current task only when relevant. Failed Model Readiness Checks should hard-block Research Runs in v1.

## User Stories

1. As a user, I want to start a bounded Research Run from a Research Question, so that I can gather varied information without creating a long-lived research project.
2. As a user, I want a Research Proposal before a run starts, so that I can understand what will be researched before spending time or model budget.
3. As a user, I want the Research Proposal to include the Research Question, Proposal Summary, purpose, Evidence Mix, Research Budget, trigger, blocking/background mode, and optional Research Brain model override, so that I can approve the right scope.
4. As a user, I want to edit all core Research Proposal fields before approval, so that the final run matches my intent.
5. As a user, I want agent-triggered research to create a proposal only, so that agents cannot silently launch long-running research.
6. As a user, I want Research Runs to require explicit approval, so that I remain in control of cost, time, and focus.
7. As a user, I want a quick model reachability check before proposal display, so that I am not offered obviously impossible research.
8. As a user, I want a full Model Readiness Check after approval and before start, so that misconfigured Tongyi serving does not produce misleading artifacts.
9. As a user tuning the Ollama Modelfile, I want a standalone research doctor command, so that I can validate model setup without creating a run.
10. As a user tuning the Ollama Modelfile, I want doctor output to include diagnostic setup hints, so that I can rebuild and reload the model until it behaves correctly.
11. As a user, I want readiness failures to hard-block v1 Research Runs, so that bad model output is treated as a setup issue rather than research evidence.
12. As a user, I want Inline Thinking removed from normal artifacts, so that Research Briefs and Source Notes stay readable and audit-focused.
13. As a user, I want raw model responses retained only in Workspace Diagnostics or Run Diagnostics, so that troubleshooting remains possible without polluting normal research artifacts.
14. As a user, I want each Research Run to have a readable date/slug/short-id identity, so that I can browse and resume runs easily.
15. As a user, I want run artifacts stored in the Workspace Research Store, so that research stays local to the project or invocation workspace.
16. As a user, I want research artifacts to be local scratch by default, so that messy intermediate investigation is not automatically treated as project documentation.
17. As a user, I want to explicitly promote a Research Brief when it should become shared documentation, so that durable project docs remain intentional.
18. As a user, I want the Research Budget to limit searches, source visits, rounds, model calls, or elapsed time, so that the run cannot become open-ended.
19. As a user, I want the Research Brain to be able to recommend stopping early, so that completed-enough research does not waste budget.
20. As a user, I want the Research Orchestrator to enforce hard budget limits, so that early-stop flexibility does not become unlimited research.
21. As a user, I want budget exhaustion to produce a best-effort Research Brief, so that I still get useful output from partial research.
22. As a user, I want a Continuation Recommendation when more research is worthwhile, so that I can decide whether to extend the budget.
23. As a user, I want the Evidence Mix to be explicit, so that deep research seeks varied source categories rather than summarizing the top search results.
24. As a user, I want the Research Brief to report which Evidence Mix categories were found, weak, or missing, so that I can judge coverage.
25. As a user, I want Source Notes to include URL, title, retrieval time, source type, citation number, and evidence snippets, so that every claim can be traced.
26. As a user, I want raw page content excluded by default, so that research artifacts remain compact and readable.
27. As a user, I want oversized sources chunked and merged into one Source Note, so that large documentation pages can still be used without overwhelming context.
28. As a user, I want failed searches, dropped sources, missing categories, and user-excluded categories recorded as Negative Evidence or coverage events, so that dead ends, exclusions, and missing categories are visible without conflating them.
29. As a user, I want Negative Evidence reflected in gaps and confidence rationale, so that missing information affects the conclusion.
30. As a user, I want the Research Brief to use numbered citations and linked source references, so that I can audit the answer quickly.
31. As a user, I want evidence separated from interpretation, so that I can distinguish sourced facts from synthesized judgment.
32. As a user, I want Research Briefs to include low/medium/high confidence with rationale, so that I can decide how much to rely on the conclusion.
33. As a user, I want Research Briefs written for both me and the coding agent, so that I can read them directly and Pi can use them in later work.
34. As a user, I want implications for Pi or the current task only when a run was agent-triggered or task-triggered, so that general human research does not become awkwardly agent-centric.
35. As a user, I want human-initiated runs to generate a self-contained Human Research View, so that I can read the final output comfortably in a browser.
36. As a user, I want Pi to print the Human Research View path rather than opening a browser automatically, so that I stay in control of my desktop.
37. As a user, I want the Human Research View derived from canonical Markdown and JSON artifacts, so that Markdown and JSON remain the source of truth.
38. As a user, I want compact Progress Digests during a run, so that I can see budget usage, current signal, gaps, and artifact location without flooding chat context.
39. As a user, I want full intermediate details kept on disk, so that the main conversation remains manageable.
40. As a user, I want limited Steering Instructions, so that I can cancel, force synthesis, or add a constraint without turning the run into a second chat.
41. As a user, I want blocking Research Runs to pause decision-blocking grilling or design work, so that important external uncertainty is resolved before proceeding.
42. As a user, I want background Research Runs to continue while non-dependent discussion proceeds, so that research does not block unrelated progress.
43. As a user, I want active v1 runs to stop on Pi shutdown and be marked interrupted, so that there are no detached orphan processes.
44. As a user, I want Manual Resume for interrupted runs, so that persisted artifacts can be reused rather than starting over.
45. As a user, I want Manual Resume to require approval of remaining or revised budget, so that continuation remains explicit.
46. As a user using grill-with-docs, I want the agent to propose research when technology feasibility is uncertain, so that design grilling can be grounded in current external evidence.
47. As a user using grill-with-docs, I want local codebase exploration to remain separate from Research Runs, so that research is reserved for external decision-relevant uncertainty.
48. As a coding agent, I want to request a Research Proposal when an architectural choice needs current external facts, so that I can avoid guessing.
49. As a coding agent, I want Research Briefs to include task implications when I triggered the run, so that I can apply the result to implementation planning.
50. As a coding agent, I want a Run Summary refreshed between rounds, so that I can provide the Research Brain with compact working memory instead of replaying the full ledger.
51. As a coding agent, I want a Claim/Evidence Ledger, so that claims, contradictions, gaps, and synthesis decisions survive context loss.
52. As a Research Orchestrator, I want Candidate Filtering before model source selection, so that the Research Brain spends context on better candidates.
53. As a Research Orchestrator, I want deterministic deduplication and simple source ranking, so that search-result noise is reduced before synthesis.
54. As a Research Orchestrator, I want to use Pi-style web search and fetch behavior in v1, so that the feature does not require new paid external APIs.
55. As a Research Orchestrator, I want to support local text, Markdown, and source-file inputs in v1, so that research can include relevant local context without full document parsing.
56. As a Research Orchestrator, I want deterministic analysis helpers only, so that counting, deduplication, and citation linking are safe and predictable.
57. As a Research Orchestrator, I want to forbid runtime-generated code execution by the Research Brain, so that v1 avoids sandbox and security complexity.
58. As a Research Orchestrator, I want provider plugins and document parsers deferred, so that v1 focuses on the lifecycle, artifacts, and research loop.
59. As a future maintainer, I want subagent integration deferred to v2, so that v1 can first prove the Research Run model and artifact contract.
60. As a future maintainer, I want a pi-native lifecycle rather than literal DeepResearch protocol emulation, so that visualization, approval, and persistence fit Pi.

## Implementation Decisions

- Build a pi-native Research Orchestrator, not a faithful literal implementation of Alibaba DeepResearch's action/observation protocol.
- Use Tongyi DeepResearch as the single Research Brain in v1. It plans queries, prioritizes sources, extracts Source Notes when invoked by the Research Orchestrator, updates findings, recommends early synthesis, and drafts the Research Brief. It does not receive direct tool access; it proposes a limited v1 set of structured research intents, and the Research Orchestrator validates and executes side-effecting operations. V1 intents are search, select sources, update findings, synthesize brief, and stop early. Source Note extraction is a required orchestrator phase after a selected source is read, not an optional Brain-directed action.
- Keep the Research Orchestrator responsible for lifecycle control, proposal approval, budgets, source access, artifact persistence, deterministic helpers, progress visualization, readiness checks, diagnostics, and guardrails. The orchestrator, not the Research Brain, performs search, fetch, local reads, artifact writes, budget accounting, and progress reporting.
- Expose a human command surface and one high-level `deepresearch` agent tool. The command supports human initiation, same-command approval for human-triggered runs, deferred approval for edited proposals, doctor diagnostics, status, steering, rendering, resume, intentional inspection of prior brief versions with stale/failed-continuation warnings, and minimal human-only promotion such as `/research promote <run-id> --to docs/research/...`. The `deepresearch` tool supports only high-level actions: `propose`, `status`, `read_brief`, `render_view`, and `recommend_resume`. `read_brief` returns `brief.md` only for runs whose current status intentionally represents a readable brief, currently `completed` and `budget_exhausted`; it must not synthesize or assemble partial findings from `cancelled`, `interrupted`, `failed`, or `readiness_failed` runs. If a `failed` run has `previousBriefAvailable`, `read_brief` still refuses by default and directs the caller to `status` or explicit human inspection. `render_view` is likewise limited to `completed` and `budget_exhausted` runs, and refuses `failed` runs even when a previous valid brief exists. The `deepresearch` tool must not approve, deny, start, resume, cancel, force synthesis, or add Steering Instructions; those remain human command actions.
- Agent-triggered use is proposal-only until the user approves, and requires a valid Research Trigger; routine lookup, local codebase exploration, and curiosity-only requests should be refused with guidance to use normal search/fetch or local tools instead. Human-initiated use creates a file-backed proposal and asks for approval in the same command fast path, while also supporting later explicit approval for edited or deferred proposals; weak triggers may warn but can proceed after explicit confirmation. The same-command approval prompt offers explicit approve now, edit/defer, and deny paths; approval re-reads the file immediately, and editing resumes through an explicit later command rather than hidden file watching. Unapproved proposals live under the Workspace Research Store as lightweight proposal artifacts, not Research Runs; approval creates a Research Run directory and carries the approved proposal into it.
- Research Proposals are editable before approval. Editable fields include the Research Question, Proposal Summary, purpose, Evidence Mix, Research Budget, blocking/background mode, trigger, and optional Research Brain model override. The `proposal.md` file is the source of truth at approval time; approval re-reads and validates the file from disk, and parse or validation errors block the run with actionable feedback. Proposal generation happens before approval and cannot consume an approved Research Budget. V1 proposal generation is mostly deterministic: parse the user or agent request, require or derive a Research Trigger, suggest a conservative Evidence Mix and Research Budget from templates, and write editable `proposal.md`. The Research Brain is not required to draft a proposal; quick reachability verifies that the configured brain is basically available before offering a run, not because it is needed to write the proposal. If model calls are optionally used to draft proposals, they are clearly surfaced and accounted as proposal diagnostics or metadata rather than run budget.
- A quick reachability check runs before showing or creating a normal approvable proposal for both human-initiated and agent-triggered proposals, even though proposal generation is mostly deterministic. If quick reachability fails, Pi does not create a normal approvable proposal or Research Run; it returns a setup-blocked message with `/research doctor` guidance. It may optionally write a lightweight workspace diagnostic artifact such as `.pi/research/diagnostics/reachability-<timestamp>.json`, but that artifact is not a Research Proposal or Research Run. V1 does not include a draft-only bypass for unreachable Research Brain setup. After approval, Pi creates the Research Run directory. If no other run is active, it runs the full operational Model Readiness Check inside that run and writes diagnostics immediately before source work begins. If another run is active, the approved run remains `queued`; queued runs do not consume research budget, source budget, or model-call budget, and their full readiness check waits until they are about to become active because model setup can change while queued. Failure blocks research execution and leaves the run in `readiness_failed` status so the user has a stable artifact path for troubleshooting and later retry or resume.
- The operational readiness probes should validate reachability, stop-token behavior, structured-output recoverability after Inline Thinking and fence stripping, and evidence-grounded synthesis from supplied fake Source Notes.
- The doctor command runs readiness diagnostics on demand in the active Pi session workspace and emits setup hints for model/template/stop-token failures. V1 uses the configured Research Brain model by default, while allowing an explicit model override such as `/research doctor --model <id>`. Doctor output and artifacts state the exact provider, model, and prompt/template settings tested. Doctor artifacts are written under the workspace-local `.pi/research/diagnostics/` by default, while command output may still reference model/provider/global config hints.
- Inline Thinking is stripped from normal outputs. Raw model responses are retained only in Workspace Diagnostics or Run Diagnostics.
- Normal artifacts are canonical Markdown and JSON-family files. The Human Research View is derived and self-contained, not canonical. It is generated from `brief.md` plus canonical JSON state such as `status.json`, `budget.json`, `evidence-mix.json`, Source Note metadata, and brief-version metadata, so JSON state drives labels, warnings, source links, budget and coverage summaries, and stale or best-effort banners.
- Human-initiated runs auto-generate the Human Research View when they finish with `completed` or `budget_exhausted` status. Agent-triggered runs can render it on demand only for `completed` or `budget_exhausted` runs. Pi prints the path instead of opening it. For `budget_exhausted` runs, the view must clearly label the result as budget-exhausted and best-effort, include caveats and continuation recommendation when present, and visually distinguish it from a fully completed run. For `completed` and `budget_exhausted`, `view/index.html` renders the current brief and may link prior versions. For `failed` runs with `previousBriefAvailable`, normal `render_view` refuses like `read_brief`; a human-only inspection view may be generated only with explicit stale/failed-continuation labeling. Incomplete runs expose status, run summary, ledger, and source-note paths rather than a polished Human Research View.
- Research Run identity uses date, slug, and short ID.
- Research artifacts live in the workspace-local research store under the active Pi session current working directory, not the global Pi agent directory.
- Research artifacts are scratch by default. A Research Brief becomes project-shareable only through explicit promotion. V1 includes a minimal human-only promotion command; agents may recommend promotion but not perform it. Promotion writes only inside the active workspace by default, refuses paths outside the current working directory unless a future explicit escape hatch is added, creates parent directories as needed, avoids overwriting existing files unless `--force` is provided, and prints all written paths. Promotion creates a shareable package, not just a raw copy of `brief.md`: at minimum it includes the Research Brief plus a source-reference appendix with citation metadata and snippets needed to audit claims. It removes or converts local `.pi/research` paths to durable relative documentation paths and does not promote raw diagnostics, raw model responses, or full raw fetched content by default. Promotion is allowed for `completed` and `budget_exhausted` briefs, but promoted `budget_exhausted` briefs must preserve best-effort/budget-exhausted status, caveats, gaps, confidence rationale, and continuation recommendation when present. Promotion may create research documentation suitable for ADR links, but it does not automatically create or modify ADRs.
- The v1 Workspace Research Store layout uses `.pi/research/proposals/<date-slug-id>/` for unapproved proposals and `.pi/research/runs/<date-slug-id>/` for approved Research Runs. Proposal artifacts include `proposal.md`, parsed `proposal.json`, and `status.json`. Run artifacts include `proposal.md`, `question.md`, `status.json`, `budget.json`, `evidence-mix.json`, `progress-digest.md`, `run-summary.md`, `ledger.jsonl`, compact filtered candidate sets under `candidates/`, `source-notes/` JSON and Markdown files, latest-current `brief.md`, versioned `briefs/` Markdown files, optional `view/index.html`, and `diagnostics/` files such as readiness results, raw responses, parser failures, and optional raw unfiltered search dumps when diagnostics are enabled. Before approval, `proposal.md` is the editable source of truth; `proposal.json` is only a generated parse/cache for preview, validation, and tests, and must be regenerated from `proposal.md` rather than overriding it. At approval, Pi re-parses `proposal.md`, writes the approved `proposal.json`, and carries both into the run directory. After approval, run JSON and Markdown artifacts are canonical state, while `proposal.md` remains the human-readable approved request.
- Status records lifecycle state, timestamps, trigger, blocking/background mode, termination reason, interruption state, and resume metadata. Proposal statuses are `draft`, `denied`, and `approved`. Research Run statuses are `queued`, `readiness_failed`, `running`, `synthesizing`, `completed`, `budget_exhausted`, `cancelled`, `interrupted`, and `failed`. For the one-active-run limit, `running` and `synthesizing` count as active; `queued`, `readiness_failed`, `interrupted`, `cancelled`, `budget_exhausted`, `completed`, and `failed` do not. `queued` may occur briefly after approval or while another run is active. `completed` may have termination reason `normal` or `forced_synthesis`; forced-synthesis briefs must state user-forced termination, unsatisfied Evidence Mix items or gaps, and confidence impact. `budget_exhausted` is reserved for hard budget exhaustion and can still include a best-effort `brief.md`; `failed` means no trustworthy Research Brief was produced.
- Budget records approved limits, usage, and budget approval history. The orchestrator enforces the effective remaining budget even if the Research Brain wants to continue. Budget usage separately tracks searches, fetch attempts, successful source visits, extraction calls, synthesis calls, model calls or rounds, retry attempts, and elapsed time. One URL or local file counts as one successful source visit even if chunked, while failed fetches consume fetch-attempt budget and are recorded as Negative Evidence but do not count as successful source visits. Retry attempts are separately limited so bad URLs cannot loop indefinitely. Chunk extraction still consumes model-call or round budget as applicable. Readiness checks are setup validation, accounted separately in diagnostics, and do not consume the approved Research Budget; research budget covers research work after readiness passes, including planning, extraction, findings updates, and synthesis. Resume approval records a new budget allocation with timestamp, reason, and approved limits rather than overwriting the original allocation. Every budget approval or revision is also recorded as an append-only `ledger.jsonl` event for audit history, including the previous effective remaining budget, new allocation, timestamp, approval mode, and reason.
- Evidence Mix records intended categories and later marks categories as found, weak, or missing.
- The Claim/Evidence Ledger is append-only and file-based. It records claims, supporting evidence, contradictions, dropped sources, source selection decisions, gaps, steering instructions, and synthesis decisions. Source selection events record the filtered candidate set identifier, selected candidates, skipped material candidates when relevant, reasons, and budget impact. Material contradictions are first-class ledger events and must be surfaced in the Research Brief with sourced facts, conflicting evidence, interpretation or likely explanation, and decision implication; confidence is reduced unless the contradiction is resolved by stronger or more recent evidence.
- The Run Summary is refreshed between rounds and is the compact model-facing working memory shown to Tongyi. It carries question, budget remaining, evidence mix status, supported claims, contradictions, recent source notes, stable Source Note IDs, citation numbers, and unresolved gaps. It may share source data with the Progress Digest but is not optimized for human display. Tongyi may reference citation numbers in draft synthesis, while final citation formatting, validation, and linking remain orchestrator-owned.
- Progress Digests are compact, human-readable, and user-facing. They should show budget usage, current source coverage, current signal, unresolved gaps, next step, and artifact pointers without dense model-working-memory detail.
- Source Notes are extracted, goal-relevant, citation-numbered, and traceable. They include metadata and evidence snippets but not raw full source content by default. Web Source Notes record URL, final URL, title when available, retrieval time, source type, fetched content type, truncation flag, citation number, and relevant snippets. Truncated fetched content can support claims only about the retrieved portion; if truncation may affect the Research Question, the orchestrator should fetch or chunk more within budget or record a gap/Negative Evidence, and the Research Brief should mention truncation when it materially limits confidence. Local-file Source Notes record path, retrieval time, source type, citation number, relevant snippets, and a content hash so later file changes are detectable. Raw fetched content may be retained only in Run Diagnostics when needed.
- Oversized fetched content is chunked. Tongyi extracts chunk-level notes, and the orchestrator merges them into one Source Note. If chunk extraction partially fails, the merged Source Note is allowed only when successful chunks contain relevant reliable evidence, and it must mark partial extraction while diagnostics and ledger events list failed chunks. If a failed chunk may contain relevant missing context, the run records a gap and reflects confidence impact. If no chunks produce reliable notes, no Source Note is created and extraction failure is recorded as Negative Evidence and diagnostics.
- Candidate Filtering deterministically deduplicates, annotates, and applies coarse ranking or downranking rules before Tongyi selects sources. It prefers official, primary, recent, and high-signal sources when relevant, downranks low-signal or SEO-heavy pages, and preserves traceability metadata. In v1 it may drop exact or near duplicates, inaccessible or invalid URLs, obvious spam or SEO farms, and sources excluded by proposal or steering; otherwise it prefers downranking over dropping. Filtered candidate sets are stored as compact JSON artifacts such as `candidates/search-0003.json`, containing query, retrieval time, filtered and annotated candidates, ranks, filtering reasons, and dropped-candidate summaries. Raw unfiltered search dumps are not stored by default unless diagnostics are enabled. Dropped candidates are recorded with reasons when material to coverage. Tongyi selects among filtered candidates with reasons and does not receive the raw unfiltered search dump in v1.
- Web Research Access uses Pi-style search and fetch in v1. New paid providers are not required for v1.
- Local text, Markdown, and source-file inputs are supported as supporting context when the primary trigger is external decision uncertainty. Local-source notes are cited separately from web sources. Full PDF, office document, audio/video, and archive parsing is deferred.
- The Research Brain may recommend early synthesis. The orchestrator may accept this only if minimum proposal evidence requirements are satisfied, or if missing Evidence Mix categories have credible Negative Evidence recorded, and no hard blockers remain.
- Budget exhaustion produces a best-effort Research Brief with caveats and, when appropriate, a Continuation Recommendation. `budget_exhausted` is terminal for the approved budget allocation but resumable via explicit continuation. If the user approves continuation, the same Research Run can move back to `queued` or `running`. `brief.md` always points to the latest current valid Research Brief, while every produced brief is preserved under `briefs/` as a versioned artifact such as `0001-budget-exhausted.md` or `0002-final.md`. The ledger records continuation, budget revision, prior brief status, and which brief version is current; `view/index.html` renders the latest valid brief and may link prior versions. If continuation synthesis fails after a prior valid brief exists, the orchestrator does not overwrite `brief.md`; it preserves the previous current brief, appends a failed synthesis ledger event, writes diagnostics, and sets status to `failed` with metadata such as `previousBriefAvailable` and a pointer to the last valid brief. `read_brief` refuses `failed` runs by default even when a previous valid brief exists, and points users or agents to `status` or an explicit human inspection command, such as `/research brief <run-id> --previous` or `/research inspect <run-id>`, so stale evidence is not accidentally treated as current. Human inspection commands must clearly label prior brief versions as stale when later continuation failed.
- Limited steering is supported: cancel, force synthesis after current step, and add an instruction. Full interactive replanning is not v1. `cancel` stops without producing a Research Brief, writes final status/progress summary with artifact paths and reason, and preserves source notes and ledger; if the user wants best-effort synthesis, they should use `force_synthesis`, or the cancel flow may offer “cancel without brief” versus “synthesize then stop.” `force_synthesis` is refused when no Source Notes exist, because that would produce unsupported model opinion; with some Source Notes but unmet evidence minimums, it may produce a caveated forced-synthesis brief. `add_instruction` may narrow, prioritize, exclude, or clarify within the approved Research Question, but may not broaden scope, add a substantially new comparison axis, or require a new Evidence Mix without explicit continuation or a new Research Proposal. Every Steering Instruction appends a `ledger.jsonl` event with timestamp, instruction type, user-provided text when present, budget state at the time, whether it was applied/rejected/deferred, and how the orchestrator applied it. The final Research Brief mentions steering when it materially affected coverage, confidence, or termination.
- Blocking Research Runs pause the dependent decision path. Background Research Runs are in-process asynchronous runs owned by the current Pi session; they emit Progress Digests, respond to status checks, and complete without blocking unrelated conversation. They are not detached processes; if in-process concurrency is unavailable, v1 should degrade to explicit polling or step execution rather than detached execution. V1 allows multiple proposals and historical runs but only one active Research Run per Pi session/workspace; additional approved runs remain `queued` or ask the user to cancel or finish the active run first.
- On Pi shutdown, v1 active runs stop and are marked interrupted. Detached background execution is out of scope for v1.
- Manual Resume covers `interrupted`, `readiness_failed`, and `budget_exhausted` runs. For `readiness_failed`, it shows readiness diagnostics and reruns readiness after approval before any source work begins. For `interrupted`, it shows completed Source Notes, ledger state, budget used and remaining, and asks the user to approve remaining or revised budget before continuing. For `budget_exhausted`, continuation requires explicit additional budget approval and preserves prior brief versions. `completed` runs are terminal in v1; new facts or angles require a new Research Proposal, optionally referencing the old brief as local context.
- Research Briefs are decision-supporting but not agent-only. They include bottom line, confidence, evidence, interpretation, tradeoffs, caveats, citations, source list, gaps, and optional continuation recommendation. Factual claims in a Research Brief must be supported by Source Notes extracted from fetched/read source content or local inputs; search result snippets and candidate metadata may guide selection or Negative Evidence but must not be cited as factual evidence.
- Research Briefs include implications for Pi or the current task only when agent-triggered or task-triggered.
- Citations use numbered references and linked source entries in Markdown and HTML. The orchestrator validates that every Research Brief citation references an existing Source Note and, where feasible, that cited Source Notes are relevant to the nearby claim. Invalid citations trigger repair or retry within the approved synthesis/model-call budget, and citation repair attempts are limited by the Research Budget. If budget is exhausted during citation repair, synthesis fails unless a valid prior brief version exists; invalid-citation briefs must not be emitted as current `brief.md`. When a prior valid brief exists, it remains the latest current brief and status metadata must make the failed later synthesis explicit.
- Deterministic Analysis Helpers may perform fixed operations such as deduplication, counting, table normalization, budget accounting, and citation linking. Model-written runtime code execution is forbidden in v1.
- The Research Brain model is resolved from an approved proposal override first, then extension configuration, then an environment variable fallback, then a documented conventional default only as a last resort. Any run-specific model override must be present in the proposal and explicitly approved by the user. The full readiness check tests the exact resolved model for the run immediately before source work begins. The resolved provider/model/template settings are written into proposal and run metadata so later diagnostics show exactly what was used. `models.json` remains provider/model catalog metadata, not the place where this extension’s default Research Brain is chosen. The Tongyi model entry should keep provider reasoning disabled if the provider does not support a formal reasoning channel. Inline thinking is handled as plain text normalization, not provider reasoning.
- The current basic Modelfile setup is not assumed to be ready. Readiness checks and doctor diagnostics are part of the product, not optional developer conveniences.

## Testing Decisions

- Tests should validate external behavior and artifact contracts rather than private implementation details.
- The proposal manager should be tested for file-backed proposal creation, editable-field parsing, approval/denial behavior, and agent-triggered proposal-only enforcement.
- The readiness checker should be tested with mocked model responses for pass/fail cases: unreachable model, runaway output, missing stop behavior, malformed structured output, recoverable fenced JSON, recoverable Inline Thinking, and failed evidence-grounding.
- The model response normalizer should be tested for stripping Inline Thinking, removing Markdown fences, extracting JSON, preserving diagnostics, and refusing unrecoverable structured output.
- The run store should be tested for creating readable run identities, writing canonical artifacts, appending ledger entries, updating status, and marking interruption.
- The budget module should be tested for usage accounting, hard-limit enforcement, early-stop acceptance, budget-exhausted brief flow, and continuation recommendation behavior.
- Candidate Filtering should be tested for URL deduplication, primary-source preference, low-signal downranking, and metadata preservation.
- Source access should be tested with mocked search/fetch responses and local text inputs rather than depending on live network calls in unit tests.
- Source-note extraction should be tested as an orchestrator-owned phase with mocked model outputs for normal sources, oversized chunked sources, merge behavior, citation numbering, and traceability metadata.
- The Claim/Evidence Ledger should be tested for append-only event recording and resumable reconstruction of claims, contradictions, gaps, source selection decisions, dropped sources, and user-excluded categories distinct from searched-but-missing categories.
- The Run Summary builder should be tested to ensure it includes compact working memory and does not replay the full ledger.
- The Progress Digest renderer should be tested for budget usage, evidence mix status, current signal, gaps, and artifact pointers.
- The Research Brief renderer should be tested for required sections, numbered citations, citation validation, evidence-versus-interpretation separation, confidence rationale, gaps, and optional implications.
- The Human Research View renderer should be tested as self-contained HTML derived from `brief.md` plus canonical JSON state, including status labels, warnings, source links, budget and coverage summaries, stale/best-effort banners, and no external assets.
- Steering should be tested for cancel, force synthesize, and add instruction. It should not support unrestricted parallel chat in v1.
- Manual Resume should be tested for interrupted-state discovery, previous-state presentation, revised budget approval, and continuation without repeating completed Source Notes.
- Shutdown handling should be tested for marking active runs interrupted and avoiding detached orphan processes.
- Command and tool entrypoints should be tested for human versus agent behavior, especially that agent-triggered use cannot silently start a run.
- Prior art exists in the current extension tests for subagent lifecycle, stream/progress processing, command construction, activity-feed rendering, and permission policy behavior. New tests should follow the same style of small modules with mocked boundaries.
- Integration tests should cover a minimal end-to-end Research Run with mocked model/search/fetch components: proposal, approval, readiness, search, source note, ledger update, run summary, brief, and human view.
- Live model tests should be kept as explicit diagnostics or opt-in checks, not normal CI, because they depend on Ollama availability and local model setup.

## Out of Scope

- Faithful v1 emulation of Alibaba DeepResearch's literal action/observation protocol.
- Subagent integration for running research steps.
- Detached background processes that keep running after Pi exits.
- Automatic Research Run starts without user approval.
- User override of failed full Model Readiness Checks.
- Long-lived Research Projects or a reusable knowledge base.
- Global research storage by default.
- Automatic promotion of Research Briefs into shared project documentation.
- Automatic browser opening for the Human Research View.
- Mandatory paid search or fetch APIs.
- Full document parsing for PDFs, office documents, archives, audio, or video.
- Browser automation by default.
- Runtime-generated or model-written code execution.
- Numeric confidence scoring.
- Full interactive steering or a second research chat interface.
- Optimizing the Research Brief exclusively for agent consumption.
- Including Inline Thinking in normal artifacts.

## Further Notes

- This PRD follows the glossary in the Pi Agent Research context and the ADR choosing a pi-native Research Orchestrator.
- Alibaba DeepResearch remains valuable as process inspiration: ReAct-style iteration, batch search, goal-directed source extraction, token/budget circuit breakers, JSONL run artifacts, and final answer demarcation. V1 adapts those ideas into Pi-native lifecycle and artifact contracts rather than copying the protocol.
- Live testing of Tongyi through Ollama showed that a basic prompt template can produce malformed or runaway responses. The product must include readiness diagnostics and response normalization from the start.
- The first implementation should favor deep, testable modules around lifecycle, artifacts, readiness, and rendering. Search provider quality and richer parsing can improve later without changing the Research Run domain model.
